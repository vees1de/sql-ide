Правильное решение на основе
быстрой аналитики
SQL IDE для non-tech аудитории:
self-service аналитика на естественном языке

Цель кейса
Разработать прототип интеллектуальной
платформы, которая позволяет
пользователям получать данные из базы
данных без знания SQL, используя
естественный язык.
Спросил → получил данные
→ принял решение.
Главная задача — сделать аналитику
доступной каждому бизнес-
пользователю.
Почему возникает проблема:
– Менеджеры задают вопросы на бизнес-языке
данные хранятся в формате SQL
– Требуется помощь технических специалистов
– Получение данных занимает часы или дни
– Решения принимаются без актуальной
информации
Идея кейса — устранить этот разрыв.

Проблема: данные есть — ответов нет
Сегодня в компаниях данные есть почти у всех,
но пользоваться ими могут не все.
СИТУАЦИЯ СЕЙЧАС
Менеджер Техн.спец
Представьте ситуацию:
Менеджеру СРОЧНО нужно узнать показатели продаж,
маркетологу — эффективность рекламы,
руководителю — динамику выручки.
Он задаёт вопрос техническому специалисту.
Специалист занят.
Отчёт приходит через день или два.
К этому моменту решение уже принято — без данных.
ИТОГ: потрачено время двух людей, результат работы = 0,
а самое главное — решения принимаются «вслепую».

Ожидаемое решение
В рамках кейса необходимо создать работающее решение, которое:

1. Принимает запрос на естественном языке
2. Автоматически генерирует SQL-запрос
3. Выполняет его к базе данных
4. Показывает результат в виде таблицы или графика
5. Позволяет сохранить отчёт для повторного использования.
   «Спросил → Получил данные → Увидел график → Сохранил отчёт».
   Такой подход позволяет бизнесу самостоятельно получать нужные
   данные за считанные секунды без знания SQL, снижая зависимость
   от технических специалистов. Это ускоряет принятие решений
   и позволяет использовать данные тогда, когда они действительно
   актуальны.

Дополнительные критерии оценивания
Критерии оценки технических экспертов:

1. Ценность решения для бизнеса (0–15 баллов).
   Насколько хорошо команда закрывает проблему
   долгого пути от вопроса до ответа и снижает
   зависимость от техн.специалистов.
2. Качество MVP и реализуемость (0–20 баллов)
   Есть ли рабочий прототип, который реально проходит
   путь: текстовый запрос → SQL-результат →
   визуализация.
   3.Точность NL → SQL интерпретации (0–20 баллов)
   Точность интерпретации natural language. Насколько
   корректно система понимает метрики, фильтры,
   периоды, группировки, сравнения и так далее.
3. Корректность SQL и работы с данными (0–15 баллов)
   Устойчивость к ошибкам, прозрачность логики запроса.
4. Безопасность и guardrails (0–15 баллов)
   Наличие проверок SQL, ограничений доступа, защиты
   от опасных и некорректных запросов.
5. UX / explainability / визуализация (0–10 баллов)
   Понятно ли, как система трактовала запрос, легко ли
   получить и сохранить отчет.
6. Качество демо и ответов на вопросы (0–5 баллов)
   Насколько убедительно показан сценарий использования
   и есть ли понятный путь масштабирования после MVP.

Дополнительные критерии оценивания
Бонусные баллы до +20: 8. Сохранение отчета и расписание рассылки (0–5 баллов). 9. Семантический слой или словарь бизнес-терминов (0–5 баллов). 10. Обработка неоднозначных запросов / confidence score (0–5 баллов). 11. Шаблоны типовых вопросов или переиспользуемые сценарии (0–5 баллов).
